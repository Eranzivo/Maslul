# Compliance - תאימות, פרטיות ואבטחה

> שכבת **Compliance** (תאימות; עקרונות SOC 2 / GDPR / RBAC / הצפנה). מותאמת לעולם הנדל"ן בישראל -
> **חוק הגנת הפרטיות** ועקרונות **GDPR**, ולא HIPAA (זה אינו תחום בריאות).

## מסגרת

| עיקרון | יישום |
|---|---|
| **מינימיזציה** | אוספים רק את המידע הדרוש למשימה; לא מבקשים ת"ז/פרטי תשלום אם אין צורך |
| **הגבלת מטרה** | מידע שנאסף לפתיחת קריאת שירות לא משמש למטרה אחרת |
| **RBAC** | גישה לפי תפקיד (ראו [`../guardrails/permissions.md`](../guardrails/permissions.md)); דייר רואה רק את שלו |
| **הצפנה** | במנוחה ובתעבורה (in transit / at rest) - ברמת הפלטפורמה |
| **בידוד בין משתמשים** | אין דליפת מידע בין דיירים/פרויקטים; כל שליפת זיכרון מסוננת לפי המשתמש |
| **הסכמה** | לפני שיתוף פרטים אישיים בהסלמה - נדרש `consent_to_share` |
| **שמירה ומחיקה** | מדיניות שמירה מוגבלת בזמן; זכות עיון ומחיקה (בהתאם לחוק) |
| **ללא אימון על מידע אישי** | מידע אישי של דיירים לא משמש לאימון מודלים |

## טיפול ב‑PII (מידע אישי)

- **לא לבקש** מספרי ת"ז, פרטי אשראי, או מסמכים רגישים אלא אם הכרחי לפעולה ספציפית ומאושרת.
- אם המשתמש שולח PII באופן יזום - להשתמש רק לצורך המשימה, לא לחזור עליו בטקסט שלא לצורך, ולתעד במינימום הנדרש.
- בהסלמה - להעביר PII רק ליעד המורשה ורק לאחר הסכמה.

## מה לעולם לא נשמר בזיכרון

- מספרי ת"ז / אמצעי תשלום מעבר לנדרש.
- כל מידע ללא הסכמה.
- "מסקנות משפטיות מחייבות" - הסוכן המשפטי אינו פוסק, ולכן אין "פסיקה" לשמור.

## חיבור ל‑Guardrails

ה‑Compliance עובד יחד עם שכבת ה‑Guardrails: סינון פלט מונע **דליפת PII** ו**הבטחות יתר**, וה‑Observability
שומר audit trail לכל החלטה רגישה. ⚑ יישום מלא (אישורי SOC 2 וכו') הוא ברמת הפלטפורמה - כאן מוגדרת המדיניות.
