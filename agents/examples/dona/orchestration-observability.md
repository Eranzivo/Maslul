# Observability - נראוּת ובקרה בזמן אמת

> שכבת **Real‑Time Observability**. תפקיד: כל שיחה, החלטה והסלמה - מתועדות, ניתנות למדידה ולשיפור
> (Resolved, Awaiting human, שביעות רצון).

## מה מתעדים (לכל אינטראקציה)

- מזהה שיחה, ערוץ, שפה, תפקיד משתמש, פרויקט (אם ידוע).
- כוונה שזוהתה + רמת ביטחון + הסוכן שנבחר.
- מקורות הידע ששימשו לתשובה (source + version) - לצורך **עקיבות ואנטי‑הזיה**.
- האם נפתרה ע"י הסוכן או הוסלמה (**bot‑vs‑human resolution**), ולמי.
- דחיפות (P1–P3), זמן תגובה, שביעות רצון (אם נמדדת).

## מדדי ליבה (KPIs ל‑CSM/onboarding)

| מדד | למה חשוב |
|---|---|
| **% נפתר ע"י הסוכן** (Resolved by agent) | ערך אוטומציה; יעד דונה - להפחית עומס מהמוקד |
| **% הוסלם לאדם** + פילוח לפי סיבה | מזהה פערי ידע/מקרים לשיפור הפרומפט |
| **שביעות רצון לקוח** | חוויית "דונה עם הפנים לדייר" |
| **זמן עד מענה / עד הסלמה** | תיעדוף לפי דחיפות |
| **Top intents** | מה הדיירים הכי שואלים → מה להוסיף לבסיס הידע |
| **קריאות שירות שנפתחו** | חיבור CRM, מעקב follow‑up |

## Audit trail

לכל **החלטה רגישה** (הסלמה, פתיחת קריאת שירות, החלטת פיצוי, סירוב/דיסקליימר) נשמר רישום בלתי‑ניתן‑לשינוי:
מי/מה החליט, מתי, על סמך איזה מידע, ומה היה הפלט. זה המנגנון שמאפשר אחריותיות (accountability) ושיפור מתמשך,
ותומך בדרישות ה‑Compliance.

## לולאת שיפור (מחבר ל‑QA)

תובנות מ‑Observability מזינות את חבילת ה‑Evals (`evals/`): intent עם הרבה הסלמות → להוסיף ידע/דוגמאות
לפרומפט; טעות שחזרה → להוסיף בדיקה אדברסריאלית שתתפוס אותה ב‑build בעתיד.
