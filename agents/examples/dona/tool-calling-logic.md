# לוגיקת קריאת כלים ואינטגרציות (Tool Calling)

> כלים ואינטגרציות (CRM, WhatsApp, Booking). עקרון: **סט כלים קטן,
> מתואר היטב, החזרות מובנות ומעוגנות, ואישור לפני כל כתיבה.** הרשאת כלי נגזרת מהתפקיד (RBAC).

## עקרונות תכנון (best practice)

1. **מינימליזם:** רק כלים שמשרתים את המשימה; אין כלי "ליתר ביטחון".
2. **תיאור מדויק:** תיאור הכלי חשוב כמו הפרומפט - מתי לקרוא, מה הקלט, מה הפלט.
3. **קריאה מול כתיבה:** כלי **קריאה** (lookup/search) - חופשי; כלי **כתיבה** (open/schedule/draft) - **דורש
   אישור מפורש של המשתמש** ולעיתים אישור אנושי (Supervisor).
4. **Grounding:** כלי ידע מחזירים תוכן **עם מקור+גרסה**; הסוכן מצטט אותם.
5. **Fail‑open לבטיחות:** כשל כלי → "אין לי מידע מאומת" + הסלמה, לא המצאה.
6. **שלמות לפני כתיבה:** לפני כלי כתיבה (קריאת שירות/תיאום/טיוטה) - אסוף את **כל** השדות הנדרשים; שאל על חוסרים ב**בקשה אחת מרוכזת**, כדי שהפעולה תיפתח מלאה (מונע הלוך‑ושוב ומקצר SLA). הפלט כולל סיכום + פריטי פעולה לגורם המטפל.

## כלים משותפים (שכבת התזמור)

| כלי | סוג | תיאור |
|---|---|---|
| `knowledge_search(query)` | קריאה | RAG על בסיס הידע/Vault (relevant‑only); מחזיר תוכן+מקור+גרסה |
| `get_memory(scope)` | קריאה | שליפת זיכרון מסונן לפי משתמש/תפקיד |
| `handoff(to, payload)` | פעולה | מעבר מסודר לסוכן/Supervisor/אדם עם payload |
| `detect_language(text)` | קריאה | זיהוי שפת המשתמש |
| `log_interaction(...)` | פעולה | תיעוד ל‑Observability/audit |

## כלים פר‑סוכן

### בדק בית
| כלי | סוג | הערה |
|---|---|---|
| `lookup_warranty(defect_type)` | קריאה | מחזיר תקופת בדק/אחריות **מה‑Vault** + מקור |
| `get_project_info(project)` | קריאה | מועד מסירה, מפרט, ליקויים ידועים (מסונן לפי המשתמש) |
| `open_service_call(details)` | **כתיבה→אישור** | פותח קריאה ב‑CRM לפי שדות טופס דונה (פונה + תיאור הפונה, מיקום, אזור התקלה, תחום, נושא, תיאור); zero manual entry |
| `schedule_visit(slot)` | **כתיבה→אישור** | קביעת ביקור טכנאי (booking‑from‑chat) |

### משפטי וחוזים
| כלי | סוג | הערה |
|---|---|---|
| `doc_vault_search(query)` | קריאה | רק ערכים מאושרים מה‑Vault |
| `explain_clause(clause)` | קריאה | הסבר סעיף + דיסקליימר |
| `generate_draft(doc_type, params)` | **כתיבה→אישור** | טיוטה מתבנית מאושרת, מסומנת "לבדיקת עו"ד"; **לא** חותם/מגיש |

### קבלנים וביצוע
| כלי | סוג | הערה |
|---|---|---|
| `get_project_info(project)` | קריאה | מוגבל לפרויקט שהוקצה (RBAC) |
| `get_procedure(topic)` | קריאה | נוהל/תהליך מבסיס הידע |
| `lookup_term(term)` | קריאה | מילון מושגי ביצוע |
| `schedule_coordination(request)` | **כתיבה→אישור** | בקשת תיאום עבודה מסודרת |

## אינטגרציות

- **CRM** - תיעוד אוטומטי של כל אינטראקציה/קריאה (zero manual entry, no missed follow‑ups).
- **WhatsApp** - ערוץ (+ צ'אט אתר).
- **Booking** - קביעת ביקורי שירות/בדיקה.
- **תשלומים - מחוץ לתחום בכוונה** לשלושת הסוכנים האלה. החלטת "רלוונטי בלבד", לא השמטה.
